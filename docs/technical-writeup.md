# Rudder — Technical Write-Up

## Overview

Rudder is a mobile training companion for open-water swimmers. The core premise is simple: pool training sessions are normalized to open-water race effort, giving athletes an accurate picture of their race readiness rather than raw yardage or times. The app generates AI-powered training plans personalized to each swimmer's race, fitness level, schedule, and injury history — and adapts those plans automatically as the athlete checks in over time.

## Architecture

### Frontend (React Native / Expo)

The app uses Expo's file-based router, where each file in the `app/` directory corresponds to a screen:

```
app/
  _layout.tsx           Root layout, splash screen, gesture handler
  index.tsx             Entry redirect
  (app)/                Authenticated tab navigator
    index.tsx           "The Reveal" — home screen
    arc.tsx             Training Arc visualization
    week.tsx            This Week's sessions
    profile.tsx         User settings
  (auth)/
    sign-in.tsx         Apple / Google login
  race/
    new.tsx             3-step race entry wizard
    [id].tsx            Race detail
  workout/
    [id].tsx            Workout detail + RPE logging
    manual.tsx          Manual session entry
  check-in.tsx          8-week / 4-week check-in flow
  auth/callback.tsx     OAuth redirect handler
  health/workouts.tsx   HealthKit session browser
  profile/
    baseline.tsx        Baseline fitness assessment
    benchmark.tsx       Critical Swim Speed test
  races.tsx             Race list
```

### Backend (Supabase)

Supabase handles the database, authentication, and server-side logic:

**Tables:**
- `user_profiles` — swimmer identity, preferences, injury flags, schedule
- `races` — target race details (distance, date, body of water, goal time)
- `training_plans` — full JSON training plan generated by Claude, stored per race
- `swim_sessions` — completed swim sessions (from HealthKit or manual entry)
- `check_ins` — 8-week and 4-week check-in records (ahead / on_target / behind)

**Row Level Security (RLS)** is enforced on all tables — users can only read and write their own data. A PostgreSQL trigger auto-creates a `user_profiles` row whenever a new user signs up.

### AI Layer (Supabase Edge Function)

All Claude API calls are proxied through a Supabase Edge Function (`supabase/functions/ai/index.ts`). This keeps the Anthropic API key server-side and never exposes it to the client.

The edge function handles four actions:

| Action | Description |
|---|---|
| `generate_plan` | Generates a full multi-phase training plan for a given race |
| `adjust_plan` | Makes incremental adjustments based on check-in status |
| `interpret_session` | Cleans and normalizes raw wearable session data |
| `reveal_copy` | Generates a short motivational line for the home screen |

Claude receives the full swimmer profile (experience level, available days, session duration preferences, injury flags, dominant stroke, miss tolerance, primary identity) and generates structured JSON training plans with phase blocks, week blocks, and individual planned sessions — each with intervals, effort bands, and intent descriptions.

**Key AI constraints enforced:**
- 2-week taper lock: no structural changes in the final 2 weeks before race
- Miss tolerance shapes how aggressively missed sessions are compensated
- Injury flags prevent loading flagged areas
- At 4 weeks out with "behind" status: protect confidence over fitness gains

### Authentication

- **Apple Sign-In**: uses `expo-apple-authentication` with a SHA-256 hashed nonce (via `expo-crypto`) passed to Supabase's `signInWithIdToken`
- **Google OAuth**: Supabase's built-in OAuth flow with a web browser redirect
- Sessions are persisted using a custom SSR-safe storage adapter (avoids `AsyncStorage` on web to prevent server-side rendering crashes)

### Health & Watch Integration

- **Apple HealthKit**: reads swim workouts from the Health app via `react-native-health`. Sessions are interpreted by Claude before being matched to planned sessions.
- **WatchKit**: sends planned workout intervals to an Apple Watch companion app via `react-native-watch-connectivity`. Falls back to a queue if the Watch app is not currently open.

## Training Logic

### Pool Normalization
Distances are normalized across pool lengths (20m, 25y, 25m) to ensure pace comparisons are accurate. Many wearables incorrectly assume 25m — the AI layer flags and corrects this.

### Training Phases
Plans are structured around four phases:
- **Base** — aerobic foundation, easy effort
- **Build** — volume + intensity increase
- **Specific** — race-pace and open-water simulation
- **Taper** — volume reduction, sharpening

Phase assignment is based on days to race, not a fixed calendar — so the plan always reflects where the athlete actually is relative to their race.

### RPE Logging
After each session, the athlete rates perceived effort on a 1–10 scale. This feeds the compliance tracking and informs plan adjustments.

### Check-In System
At 8 weeks and 4 weeks out, the athlete self-assesses their training status (ahead / on_target / behind). This triggers an AI plan adjustment with a plain-English explanation of what changed and why.

## Cross-Platform Development

The repo is set up for both Mac and Windows developers:

- **Line endings**: `.gitattributes` enforces LF normalization — no CRLF conflicts when switching between OS environments
- **Secrets**: `.env` is gitignored. Developers copy `.env.example` and fill in their own Supabase credentials
- **iOS builds**: Require a Mac with Xcode installed. EAS Build can also compile remotely without a local Mac
- **Install**: `npm install --legacy-peer-deps` (required due to peer dependency conflicts in this project)
